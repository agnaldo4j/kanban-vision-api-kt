# Engineering Flow Baseline — July 2026 (GAP-BI)

The project's **first flow signal** for its own delivery flow — the ISO/IEC 25010 *Evolutionary
Change* dimension had no flow-feedback layer (no lead-time / cycle-time / WIP measurement) until
now (`docs/quality/audit-2026-07-below9.md`, finding #1). Numbers are derived from the
repository's **merged-PR history** via `scripts/flow-metrics.sh` — re-run each cycle and update
this snapshot (same window definition) so comparisons stay honest.

> This is the engineering-process analogue of the k6 performance baseline
> (`performance-baseline-2026-07.md`, ADR-0027): a reproducible "first signal", not a gate.

## Method

| Item | Value |
|---|---|
| Data source | GitHub merged-PR history (`gh api graphql`) + Project #6 board |
| Tool | `scripts/flow-metrics.sh` (gh GraphQL + python3) |
| Window | last **40 merged PRs**, `#225..#265` |
| Merged span | 2026-07-05 → 2026-07-09 |
| Date measured | 2026-07-09 |

- **PR cycle time** = merge time − PR open time (review/integration latency).
- **Lead time** = merge time − first-commit time (dev + review latency).
- **Cadence** = merged PRs per week over the window.
- **PR size** = additions + deletions (ties to the GAP-BH `pr-size` soft-gate).

## Results

| Metric | p50 | p85 | max |
|---|---|---|---|
| PR cycle time (open→merge, h)   | **0.3** | 0.9 | 58.5 |
| Lead time (1st commit→merge, h) | **0.3** | 0.9 | 18.0 |
| PR size (changed lines)         | **100** | 330 | 3785 |

- **Delivery cadence:** **70.9 PRs/week** (40 PRs over 3.9 days).
- **PRs over the 400-line heuristic:** **5/40** — empirical evidence for why GAP-BH added the
  `pr-size` soft-gate; the max (3785) is an outlier deletion/refactor PR.
- **WIP snapshot (board #6):** Doing = **1** (limit respected), Todo = **0**.

> Reading note: the window covers a concentrated solo-work burst (the P7 + below-9 audit cycles),
> so **71 PRs/week is a burst rate, not a steady-state cadence** — interpret cadence relative to
> the window, exactly as the k6 baseline is interpreted relative to its environment. Cycle time is
> sub-hour because merges are solo and fast; the max reflects a PR that sat open across a day.

## Product-side flow signal

Complementing this engineering-flow snapshot, the Grafana dashboard
(`observability/grafana/dashboards/kanban-vision-api.json`) now has a **"Card Completion
Throughput (cards/s)"** panel over `kanban_card_completed_total` — the product's first flow signal,
alongside the existing "Simulation Throughput" panel.

## How to reproduce

```bash
# Default window (last 40 merged PRs):
scripts/flow-metrics.sh

# Custom window (e.g. last 60 merged PRs):
scripts/flow-metrics.sh 60
```

Paste the output into this file under the same window definition, then commit the refreshed
snapshot as part of the cycle.

> Policy: this flow snapshot is an executable, **non-blocking** signal refreshed each cycle via
> `scripts/flow-metrics.sh` — never a PR gate. Changing the window or metrics requires a new
> documented measurement here (mirrors the ADR-0027 measurement policy).
